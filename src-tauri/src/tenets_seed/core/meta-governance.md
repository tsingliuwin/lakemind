# 准则变更准则——准则的准则

准则体系要活下去、活得好，自身的演化也必须有规则。这是准则的"元规则"——约束准则本身如何变更。

## 1. 准则的生命周期

每条准则都经历以下阶段：

```
提议 → 验证 → 发布 → [修订] → [废止]
```

### 提议
- 任何人在使用 LakeMind 过程中遇到的**真实分析教训**，都可以提议为新准则。
- 提议必须附带**来源案例**（Citations），说明是什么场景下、犯了什么错误、造成了什么后果。

### 验证
- 提议的准则必须经过至少一次真实场景的验证——确认按照该准则操作确实能避免错误。
- 纯理论推导的"应该这样做"不够，必须有"不这样做就会出问题"的反面案例。

### 发布
- 验证通过的准则写入对应目录的 `.md` 文件，遵循 OKF v0.1 规范（frontmatter + body）。
- 新准则必须包含：`type`、`title`、`description`、`tags`、`timestamp`、`resource` 等 frontmatter 字段。
- 正文必须包含 `# Citations` 区块，记录准则的来源。

### 修订
- 已发布准则的修改同样需要附来源案例，说明为什么现有表述不够准确或不够完整。
- 修订后更新 `timestamp` 字段。

### 废止
- 如果某条准则被证明已不适用（如技术环境变化），标记为废止而非直接删除。
- 在 frontmatter 中添加 `status: deprecated` 和 `deprecated_reason`。

## 2. 禁止凭空编造准则

**每一条准则都必须基于真实的分析教训，不允许凭空想象。**

- ✅ "我们在分析教育行业数据时，把 99 元全表订单数错误地归为体验课订单，因此必须……"
- ❌ "我觉得做数据分析应该这样做，因为理论上……"

> 这条规则本身就来自真实教训：LakeMind 的核心准则（数据纪律五条红线）全部源于一次真实对话中 agent 连犯两错的事故。经过验证，这些准则确实能防止同类错误再次发生。

## 3. 准则的层级与冲突解决

准则体系有明确的层级关系：

```
总则（general-principles）
  ├─ 数据纪律红线（data-discipline）   ┐ 都是横切红线、不可触碰
  ├─ 数据安全准则（data-security）      ┘ 纪律管真实性，安全管合规
  ├─ 分则（data-profiling / data-metrics / data-cleaning / data-analysis / data-presentation）
  └─ 行业/主题准则（industry/ topic/）
```

- **下级准则不得与上级冲突。** 如果行业准则的某条建议与总则矛盾，以总则为准。
- **数据纪律红线与数据安全准则是总则之下的两条横切底线**，优先级高于分则和行业/主题准则。
- **行业准则是总则的特化，不是例外。** 行业准则只能在总则框架内补充行业特有的规范，不能推翻总则。
- **当两条同级准则冲突时**，以更具体的那条为准（特别法优于一般法）。

## 4. 种子准则与用户修改

### 种子准则（Seed）
- 种子准则随 LakeMind 软件分发，首次启动时播种到 `~/.lakemind/tenets/`。
- 种子只在文件**不存在**时写入，**永远不覆盖**用户已修改的文件。
- 软件更新带来的新种子准则会自动补充到目录中（增量播种）。

### 用户修改
- 用户可以自由编辑 `~/.lakemind/tenets/` 下的任何文件——它们是属于用户的。
- 用户修改受 OKF 目录下的 git 版本控制保护，可以回退。
- 建议用户修改时遵循本文件定义的变更规范（附来源案例），但不强制。

## 5. 新增行业/子行业准则的流程

1. **新增行业**：在 `industry/` 下创建 `<industry_name>.md` 文件（简单行业）或 `<industry_name>/index.md` + 子文件（复杂行业，有子行业）。
2. **新增子行业**：在已有行业目录下创建子文件（如 `industry/education/k12.md`），并更新行业 `index.md`。
3. **行业准则结构**：行业准则也应该有自己的"总则"（行业通用规范）和"分则"（子行业/场景特有规范），镜像核心准则的层级结构。
4. **所有行业准则必须遵循核心总则**，不得与之冲突。
