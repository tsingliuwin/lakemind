# 分则：数据画像——如何了解你的数据

> 对应总则第五条"先理解后分析"。本分则将其落实到具体的操作步骤。

## 1. 时间范围摸底

拿到任何数据源，第一件事是搞清楚**数据覆盖的时间范围**：

- **发生时间**：数据记录的业务事件发生在什么时间段？（如订单的创建时间从 2024-01 到 2026-06）
- **获取时间**：数据是什么时候从源系统导出的？导出后是否有更新？
- **时间粒度**：最小时间单位是天？小时？秒？有无时区问题？

**怎么做**：对每个时间类型字段执行 `MIN()` 和 `MAX()` 查询，确认时间跨度。注意区分"创建时间"和"更新时间"。

**常见陷阱**：
- 数据只到上个月，但用户问的是"最近的趋势"——你需要告知数据截止日期，不要假装数据是最新的。
- 发生时间和获取时间差异很大（如历史数据补录），会导致按获取时间统计出虚假的"增长"。

## 2. 数据质量评估

在分析之前，必须对数据质量有基本的了解：

### 空值率
对每列计算空值占比：`COUNT(*) - COUNT(column)` / `COUNT(*)`。

- 空值率 > 50% 的列：高度警惕，可能是可选字段或采集缺失，用它做分析前必须理解为什么是空的。
- 空值率 = 0% 的列：可能是 NOT NULL 约束，也可能是用默认值（如 0、空字符串 `''`）填充了空值——后者更危险，因为 0 和真正的零是不同的。

### 重复率
检查是否有完全重复的行：`SELECT COUNT(*) - COUNT(DISTINCT *)` 或按业务主键检查。

- 重复可能来自：数据导出重复、多次导入、JOIN 爆炸、或业务本身允许重复（如一个用户多次下单）。
- **关键**：判断重复是脏数据还是正常业务行为，再决定是否去重。

### 异常值
对数值列查看分布：`MIN / MAX / AVG / MEDIAN / PERCENTILE`。

- 极端值（如金额为负、年龄 200 岁、日期在未来）通常是脏数据。
- 但也可能是真实业务（如退款导致的负金额）——不要自动删除，先确认。

## 3. 字段含义确认

**绝不假设字段含义。** `amount` 可能是原价、实付、含税、不含税。`status` 的 1 可能是"已完成"也可能是"待处理"。

**怎么做**：
- 先看字段名和类型推测含义。
- 用 `sample_data` 看实际值，对照业务逻辑验证。
- 如果 OKF 知识库里已有字段释义（`describe_table` 返回的"业务释义"），以它为准。
- 如果没有，**问用户确认**，然后立即 `write_okf_block` 写入知识库。

## 4. 数据量级评估

了解数据的规模，影响后续分析策略的选择：

- **行数**：几百行和几百万行的处理方式完全不同。
- **列数**：列太多时需要先筛选相关列，避免信息过载。
- **存储大小**：大文件可能需要分批加载或物化缓存。

## 5. 跨表一致性检查

分析很少只用一张表。多表关联前，必须检查表与表之间的**一致性**，否则 JOIN 出来的结果会无声出错：

### 外键完整性
订单表里的 `user_id`，是不是都能在用户表里找到？如果找不到（孤立行），说明：
- 数据缺失（用户表没同步全）
- 或外键本身是脏数据

**怎么做**：`SELECT COUNT(*) FROM orders o LEFT JOIN users u ON o.user_id = u.id WHERE u.id IS NULL`——返回大于 0 就有孤立行，需要排查。

### 主键唯一性
每张表声称的"主键"是否真的唯一？导入过程中可能产生重复主键。

**怎么做**：`SELECT id, COUNT(*) FROM t GROUP BY id HAVING COUNT(*) > 1`——有结果说明主键不唯一，分析前必须处理。

## 6. 字段类型合理性识别

"类型错配"是最隐蔽的脏数据——数据看起来正常，但存的类型不对，会在计算时出错：

| 错配 | 危险 | 识别方法 |
|---|---|---|
| 金额存成 VARCHAR | `SUM(amount)` 直接报错或拼接成字符串 | `describe_table` 看 type；`SELECT typeof(amount)` |
| 日期存成字符串 | 无法按时间排序、日期函数失效 | 同上；`TRY_CAST(col AS DATE)` 失败的是脏值 |
| 手机号存成 BIGINT | 前导 0 丢失（如 010 变成 10） | 看 `MIN/MAX`，手机号不应有算术意义 |
| 布尔值存成 0/1 整数 | 容易被当数值聚合 | 结合字段名判断 |

**怎么做**：数据画像时用 `describe_table` 逐列确认类型。发现类型错配，在清洗阶段（见 `core/data-cleaning.md`）统一转换。

## 7. 数据画像检查清单

完成数据画像后，你应该能回答以下问题：

- [ ] 这份数据记录的是什么业务？覆盖什么时间段？
- [ ] 数据是什么时候获取的？是否可能过时？
- [ ] 各字段的含义我确认了吗？有没有不确定的？
- [ ] 空值率高的字段有哪些？为什么是空的？
- [ ] 有没有明显的脏数据（重复、异常值、格式错误）？
- [ ] 数据量级是多少？需要特殊处理吗？
- [ ] 如果要多表关联：外键完整性、主键唯一性检查过了吗？
- [ ] 各字段的类型合理吗？有没有金额存成字符串这类错配？

**回答不上来的问题，就是你在分析前需要先解决的问题。**
