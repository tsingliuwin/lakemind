# 分则：数据分析——从数据到洞察

> 对应总则第一条"实事求是"和第四条"最小假设"。本分则将其落实到分析阶段的具体操作。

## 1. 假设驱动分析

好的分析不是漫无目的地查数据，而是**先有问题/假设，再用数据验证**。

**标准流程**：
1. 明确要回答的问题（"体验课的转化率是多少？"）
2. 分解为可查询的子问题（"体验课有多少人？其中多少人转正价？"）
3. 确定需要的数据表和字段
4. 写 SQL 查询，获取结果
5. 基于结果得出结论

**反模式**：
- ❌ 先查一堆数据，再从中找"有意思的发现"——这叫"数据钓鱼"（data dredging），容易发现伪相关。
- ❌ 先有结论（"我觉得转化率在下降"），再找支持它的数据——这叫"确认偏误"（confirmation bias）。

## 2. 多维交叉验证

**不要只看一个维度。** 单一维度的结论往往是片面的。

**怎么做**：
- 看到"销售额下降"→ 拆分看是哪个渠道/产品/地区在降？
- 看到"转化率上升"→ 是所有用户群都在升，还是某一群人拉高了平均值？
- 看到"A 和 B 相关"→ 是否存在混淆变量 C 同时影响了 A 和 B？

## 3. 统计陷阱

### 辛普森悖论
整体趋势和分组趋势可能完全相反。例如：A 医院的两种手术成功率都高于 B 医院，但合在一起看 B 医院反而更高——因为 B 医院接了更多难度高的手术。

**怎么做**：看到聚合数据时，先按关键维度拆分确认趋势一致性。

### 幸存者偏差
你只看到了"存活"下来的数据。例如：分析"活跃用户的留存率"，忽略了大量已流失的用户——结论会过于乐观。

**怎么做**：思考"我没看到的数据是什么？缺失的那部分会改变结论吗？"

### 基数效应
小样本的百分比变化不可靠。例如：某渠道从 2 单变成 4 单，"增长 100%"——但这只是噪声。

**怎么做**：呈现百分比变化时，同时给出绝对值。样本量太小的分组，标注"样本不足，仅供参考"。

### 平均值陷阱
平均值容易被极端值拉偏。例如：10 个人的收入平均值因为 1 个亿万富翁变得毫无代表性。

**怎么做**：用中位数（MEDIAN）、分位数（PERCENTILE）辅助平均值，呈现分布而非单一数字。

## 4. 口径一致性

**分子和分母的定义必须一致。** 这是数据分析中最常见的错误来源。

- 转化率的分子和分母是"人数"还是"次数"？
- 对比两个时间段时，是"自然月"还是"过去 30 天"？
- 跨表 JOIN 时，两边的"用户 ID"是同一套 ID 体系吗？

**怎么做**：在写 SQL 之前，用文字写出指标的完整定义。例如："转化率 = 7 天内从体验课升级到正价课的去重用户数 / 完成体验课的去重用户数"。

## 5. 对比分析的规范

做对比时，确保**可比性**：

- **时间可比**：对比"本月 vs 上月"，两个月的天数是否一样？是否有节假日影响？
- **群体可比**：对比"新用户 vs 老用户"，两个群体的定义是否清晰？有无重叠？
- **条件可比**：对比"A 方案 vs B 方案"，除了方案本身，其他条件是否一致？

## 6. 时间序列陷阱

涉及时间趋势的分析，有几个经典陷阱：

### 滞后效应
原因和结果之间有时间差。比如：今天投放的广告，转化可能发生在 3-7 天后。如果你把"当日广告费"和"当日销售额"画在一起，会得出"广告无效"的错误结论——因为转化还没发生。

**怎么做**：涉及因果的时间对比，考虑滞后窗口（如广告费对齐到 7 天后的转化）。

### 同比 vs 环比
- **环比**（本月 vs 上月）：反映短期变化，但受季节性干扰大。
- **同比**（今年 7 月 vs 去年 7 月）：消除季节性，适合强季节性业务（如 K12 的寒暑假周期、零售的节假日）。

**怎么做**：强季节性业务优先看同比；季节性弱的看环比即可。不要只看一种就下结论。

### 移动平均平滑噪声
日数据波动很大（如每日订单数受周末、活动影响），直接看日趋势会被噪声淹没。用 7 日移动平均能看清真实趋势。

**怎么做**：`AVG(指标) OVER (ORDER BY 日期 ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)` 算 7 日移动平均。
