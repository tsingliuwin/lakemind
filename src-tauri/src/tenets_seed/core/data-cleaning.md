# 分则：数据清洗——从原始数据到高质量数据表

> 对应总则第五条"先理解后分析"。数据画像（`core/data-profiling.md`）发现了脏数据，本分则定义如何清洗。

## 1. 清洗的基本原则

### 保留原始，清洗另存
**永远不修改原始数据。** 清洗结果存入新的表/视图（`t_` 或 `v_` 前缀），原始数据（`s_` 前缀）保持只读。这样随时可以回溯对比、验证清洗逻辑是否正确。

### 清洗过程可追溯
每一步清洗操作都应该有明确的 SQL 逻辑，记录在 OKF 的 pipelines 中。任何人看到清洗后的数据，都能追溯到"原始值是什么、为什么这样清洗、用的什么规则"。

### 先理解再清洗
不要看到"异常"就删除。先确认它是脏数据还是正常业务行为：
- 金额为 0 → 可能是免费体验课（正常），也可能是录入错误（脏数据）
- 日期为 NULL → 可能是尚未发生的事件（正常），也可能是采集遗漏（脏数据）

## 2. 常见脏数据类型与清洗策略

### 空值（NULL）
| 场景 | 策略 | 示例 |
|---|---|---|
| 关键字段为空（如主键、订单号） | 过滤掉整行 | `WHERE order_id IS NOT NULL` |
| 可选字段为空（如备注、第二联系方式） | 保留空值，分析时注意 | 不做处理 |
| 数值字段为空但有业务默认值 | 用 `COALESCE` 填充 | `COALESCE(discount, 0)` |
| 用空字符串 `''` 伪装的空值 | 统一转为 NULL | `NULLIF(trim(col), '')` |

### 重复记录
| 场景 | 策略 | 示例 |
|---|---|---|
| 完全重复行（数据导入重复） | 用 `DISTINCT` 或 `ROW_NUMBER()` 去重 | `ROW_NUMBER() OVER (PARTITION BY id ORDER BY update_time DESC) = 1` |
| 业务允许的重复（一人多单） | 不去重，但在分析时说明统计口径 | — |
| 主键重复但内容不同（脏数据冲突） | 取最新记录或报告给用户决策 | 按 `update_time` 取最新 |

### 格式不一致
| 场景 | 策略 | 示例 |
|---|---|---|
| 日期格式混乱（`2024/1/5` vs `2024-01-05`） | 统一用 `TRY_CAST(col AS DATE)` | 失败的记为 NULL，单独排查 |
| 文本大小写不一致（`Male` vs `male` vs `MALE`） | 统一转换 | `LOWER(gender)` |
| 前后空格 | 清除 | `TRIM(col)` |
| 编码问题（乱码、特殊字符） | 尝试 `REPLACE` 或标记为异常 | 具体情况具体处理 |

### 异常值
| 场景 | 策略 | 示例 |
|---|---|---|
| 明显不合理的值（负数金额、未来日期） | 标记或过滤 | `WHERE amount >= 0` |
| 统计意义上的极端值 | 不自动删除！先确认是否正常 | 用分位数了解分布后再决策 |
| 超范围值（年龄 > 150、百分比 > 100） | 标记或修正 | 具体看业务逻辑 |

### JOIN 爆炸（极易被忽略）
多表关联后行数暴涨——通常是一对多关系被无意放大。比如订单表 JOIN 订单明细表，一个订单有 N 条明细，JOIN 后订单数看起来翻了 N 倍。

| 场景 | 策略 | 示例 |
|---|---|---|
| JOIN 后行数变多（一对多放大） | 先 `COUNT` 对比关联前后行数，确认是否被放大 | `SELECT COUNT(*) FROM a` vs `SELECT COUNT(*) FROM a JOIN b ...` |
| 需要左表粒度的指标 | 先聚合右表再 JOIN，或用子查询 | `JOIN (SELECT order_id, SUM(amount) FROM details GROUP BY order_id)` |

**关键**：任何 JOIN 之后，都要 `COUNT` 一下结果行数，和关联前对比。如果行数变了但你预期不变，就是被放大了——此时基于 JOIN 结果做 `COUNT(订单)` 会算错。

## 3. 清洗的标准流程

```
原始数据(s_) → 数据画像 → 识别脏数据 → 制定清洗规则 → 执行清洗 → 验证结果 → 输出干净表(t_/v_)
```

1. **数据画像**：按 `core/data-profiling.md` 完成。
2. **识别脏数据**：列出需要清洗的字段和问题类型。
3. **制定清洗规则**：每个问题对应一个清洗 SQL 逻辑，先用 `execute_query` 验证。
4. **执行清洗**：用 `create_view` 或 `create_table` 创建清洗后的对象。
5. **验证结果**：对比清洗前后的行数、关键指标，确保没有丢失正常数据。
6. **记录沉淀**：将清洗逻辑写入 OKF pipelines，下次遇到同类数据可直接复用。

## 4. 清洗的反模式（禁止）

- ❌ **直接在原始表上修改数据**——违反"保留原始"原则。
- ❌ **不验证就批量删除**——你删掉的可能是正常数据。
- ❌ **用 `CAST AS INTEGER` 去截断小数**——会制造假差异（见 `core/data-discipline.md` 第 4 条）。
- ❌ **看到空值就全部填 0**——0 和 NULL 的业务含义完全不同。
- ❌ **清洗了但不记录**——下次遇到同样的数据又要重来。
