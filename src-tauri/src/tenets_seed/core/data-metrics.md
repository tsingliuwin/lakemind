# 指标体系与口径管理——把指标定义清楚

> 对应总则第三条"归属正确"和第六条"分析可复现"。数据纪律红线第 2 条"禁止张冠李戴"是最危险的错误——而**多数张冠李戴的根因，是指标口径没管好**。本分则从源头预防：动手分析前，先把指标定义清楚。

## 1. 指标定义五要素——缺一不可

任何一个指标（转化率、留存率、客单价、GMV……），在分析前必须定义清楚这五个要素。定义不全的指标，算出来的数字一定有歧义：

| 要素 | 说明 | 示例（"体验转正价率"） |
|---|---|---|
| ① 名称 | 指标的唯一名称，避免歧义 | 体验转正价率 |
| ② 分子 | 满足什么条件的记录计入分子？（含筛选条件） | 7 天内从体验课升级到正价课的**去重用户数** |
| ③ 分母 | 满足什么条件的记录计入分母？（含筛选条件） | 完成体验课的**去重用户数** |
| ④ 统计窗口 | 时间范围多大？按什么时间字段？ | 按体验课**完成时间**统计，窗口为自然月 |
| ⑤ 单位与去重 | 是"人数"还是"次数"？是否去重？ | 百分比，按用户去重（一人多次只算一次） |

**怎么做**：写 SQL 之前，先用文字把这五个要素写出来（可以写在对话里给用户确认）。五要素齐全了，SQL 就水到渠成；五要素含糊，SQL 一定出错。

## 2. 口径文档化——写入 OKF 指标字典

确认后的指标定义，**立即 `write_okf_block` 写入 OKF**（如 `concepts/metrics.md`），形成团队的"指标字典"。

- **为什么**：同一个指标，不同人、不同时间可能用不同口径。写进 OKF 后，下次任何人（包括你自己开新会话）用到这个指标，都能查到统一口径，不必反复确认。
- **写什么**：指标名 + 五要素定义 + 业务含义 + 计算示例。
- **这直接落实总则第六条"分析可复现"**——口径落了地，结论才可复现。

## 3. 同名不同义陷阱——最隐蔽的口径错误

同一个词，在不同场景含义不同。对比之前必须先对齐口径：

| 指标名 | 常见的口径分歧 |
|---|---|
| 活跃用户 | 登录即活跃？还是有操作行为？DAU 按天、MAU 按月？ |
| 金额 | 原价 / 实付 / 含税 / 扣退款？不同表、不同部门可能不同 |
| 新用户 | 首次注册 / 首次下单 / 首次登录？时间窗口多长？ |
| 转化率 | 分子分母是人数还是次数？归因窗口几天？ |
| 留存率 | 次日 / 7 日 / 30 日？按注册日还是首次行为日？ |

**怎么做**：
- 看到"转化率""活跃用户"这类通用词，**先问清楚是哪一种口径**，不要凭默认理解就开查。
- 跨表对比时，确认两边的字段含义一致（如两个表的 `amount` 是不是都是实付）。
- 跨部门数据对比时，口径对齐是第一步。

## 4. 口径变更管理——变了就不能直接对比

指标口径变了（比如"活跃用户"从"登录"改成了"有操作行为"），**历史数据不能直接和现在的对比**：

- 口径变更前的数字，是按旧口径算的；变更后的是新口径——两者不可比。
- **怎么做**：必须用新口径重算历史数据，或在对比时明确标注"此处存在口径变更点，X 月前后不可直接对比"。
- 口径变更也要记入 OKF 指标字典（记录变更时间和原因）。

## 5. 指标 vs 维度——别混淆

| | 指标（Metric） | 维度（Dimension） |
|---|---|---|
| 是什么 | 算出来的数（要聚合） | 切分的视角（用来分组） |
| 例子 | 销售额、转化率、用户数 | 渠道、地区、日期、班型 |
| SQL 角色 | `SUM()`/`COUNT()`/`AVG()` | `GROUP BY` |

**常见错误**：把维度当指标分析，或把指标当维度分组。比如问"哪个渠道的转化率最高"——渠道是维度，转化率是指标，要 `GROUP BY 渠道` 算每个渠道的转化率。

## 6. 反模式（禁止）

- ❌ **不定义就用"转化率"**——分子分母窗口去重全靠猜，算出来的数字必然有歧义。
- ❌ **跨表对比不对齐 ID 体系**——A 表的 user_id 和 B 表的 user_id 不是同一套，JOIN 出来的结果全错。
- ❌ **口径变了不标注**——把口径变更前后的数字放一起对比，得出虚假的"增长"或"下降"。
- ❌ **只在自己脑子里定义口径**——不写进 OKF，下次复现时口径对不上（违反总则第六条）。
